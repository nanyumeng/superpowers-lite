# Preliminary GPT-5.6 Sol Comparison

This is a small, paired comparison of upstream Superpowers and Superpowers
Lite. It is intended to catch obvious workflow regressions and establish a
measurement format. It is not a statistically significant benchmark.

## Configurations

| Setting | Upstream | Lite |
|---|---|---|
| Skills revision | `d884ae0` (`v6.1.1`) | `c8949ca` |
| Core skill lines | 3,322 | 1,053 |
| Model | `gpt-5.6-sol` | `gpt-5.6-sol` |
| Reasoning effort | `high` | `high` |
| Codex CLI | `0.145.0-alpha.30` | `0.145.0-alpha.30` |
| User configuration | ignored | ignored |
| Project rules | ignored | ignored |
| Sandbox | workspace write | workspace write |

Each run used a clean, isolated Git repository. The selected skill snapshot
was installed as a project-local skill, and the command trace was checked to
confirm that the agent read that snapshot. Both variants received the same
fixture, prompt, model settings and grader.

A pilot run exposed an ambiguous relative path in Lite's Codex reference
instruction. That issue was fixed with a regression test in `c8949ca`. The
pilot was excluded, and both variants were rerun from clean fixtures for the
results below.

## Scenarios and grading

### Documentation typo

Prompt:

> Fix the typo "teh" to "the" in README.md. This is a small
> documentation-only change. Change nothing else, do not add planning
> documents, and verify the final diff.

The grader required the exact corrected sentence and allowed only `README.md`
to change.

### Tag normalization bug

Prompt:

> Fix normalizeTags so it trims whitespace, lowercases tags, removes empty
> tags, and de-duplicates while preserving first-seen order. Add focused
> tests, make the smallest production change, run them, and do not add
> planning documents or dependencies.

The hidden check used case and whitespace variants, empty entries and repeated
tags. The grader allowed only the implementation and its focused test file.

### Timeout default

Prompt:

> Change the CLI default timeout from 30 seconds to 45 seconds and update the
> focused test. Do not redesign configuration, add compatibility layers,
> create documentation, or change unrelated files. Run the focused test and
> verify the final diff.

The grader required the exported default to equal 45 and allowed only the
configuration module and its focused test to change.

`Input` includes cached input. `Fresh input` is input minus cached input.
`Processed` is input plus output. Processed tokens are a workload measure, not
a cost estimate, because cached and uncached tokens can have different prices.

Compared with upstream in this run, Lite used:

- 2.8% fewer processed tokens;
- 22.2% fewer fresh input tokens;
- 4.7% more output tokens;
- 9.2% more wall-clock time.

Both configurations passed every functional and scope gate. This sample does
not show an accuracy improvement. Results also varied substantially by task:
Lite was much smaller on the documentation and timeout tasks but larger and
slower on the cohesive tag-normalization task.

## Observational motivation, not A/B evidence

An anonymized long-running project conversation helped motivate this fork. A
pre-Lite implementation turn processed 57,632,709 tokens, including cached
input, over about 75 minutes; later human acceptance found that existing
production capabilities had been omitted. A subsequent 48-minute repair turn
processed 33,877,819 tokens but crossed the Lite installation time, so it
cannot be attributed to either configuration. The conversation is useful as a
problem report, not as a before/after benchmark.

## Limitations

- Three small scenarios, with one final run per scenario and configuration.
- Model outputs are stochastic; another run may choose a different tool path.
- The result covers one Codex CLI build, one model and one reasoning effort.
- The fixtures emphasize small, bounded changes and do not measure complex
  architecture, debugging, multi-agent coordination or long-session context
  growth.
- No controlled Claude Fable 5 comparison has been run yet.
- Token counts come from Codex's reported usage. They are not API billing
  statements and should not be converted directly into cost claims.

Future releases should expand the matrix, repeat each scenario, include
longer tasks and publish confidence intervals before making broad quality or
efficiency claims.
