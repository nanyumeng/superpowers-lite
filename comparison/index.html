Compatibility, not replacement

Same development spine. Lighter defaults.

Superpowers Lite starts from obra/superpowers v6.1.1 and preserves the surfaces existing projects reference. It changes the instructions that decide how much process a task receives.

What stays compatible

  • Skill names referenced by existing AGENTS.md and CLAUDE.md files.
  • The docs/superpowers/specs/ and docs/superpowers/plans/ conventions.
  • Brainstorming before new features or ambiguous scope.
  • Written plans for multi-step implementation.
  • Test-driven development for production behavior changes.
  • Systematic debugging when a problem is complex or recurring.
  • Fresh verification before declaring completion.

“Drop-in” refers to these preserved surfaces and the complete skills-directory overlay. It does not mean byte-identical behavior or a promise that every third-party harness resolves skills in the same way.

What becomes lighter

Lite removes repeated meta-instructions and makes workflow intensity conditional. Small, explicit, low-risk changes no longer require a design document and multi-step plan. Subagent dispatch, review loops, worktrees, and extended debugging remain available when the task actually benefits from them.

The design principle

Preserve guardrails that protect correctness and user control. Reduce orchestration whose primary purpose was compensating for weaker or less reliable models.

The 14 core skill instruction files are 68.3% shorter by line count than the upstream v6.1.1 baseline. That is a source measurement—not proof that shorter instructions always produce better output. See the full evidence boundary.

Which should you choose?

UpstreamChoose the original if you want the maintainer-supported behavior, prefer comprehensive ceremony by default, or cannot validate an overlay in your harness.
LiteConsider Lite if modern models are over-planning routine work and you can test the reversible overlay on representative tasks.

Start in a disposable repository. Keep the backup. Compare the same representative tasks and inspect correctness, tool calls, token use, and unnecessary changes—not speed alone.
